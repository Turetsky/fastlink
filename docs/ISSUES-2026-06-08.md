## Cross-cutting theme
**Silent failure on stale/hung content scripts.** `status` pings the relay (always healthy),
but the tab-level action channel can be dead/wedged while status says connected. Highest-value
single fix: a real per-target-tab health check that verifies the content script is live +
responsive, with auto-reinjection, so the API stops reporting success on an undrivable tab.
This is issue #1 and underlies #2/#4.

## Issue #7 ‚Äî heavy-page main-thread freeze (fix detail, 2026-06-09)

**Root cause (confirmed in `fast-ext/src/actions/page.js`).** The DOM index walk
(`buildIndexSync` / `stepInitWalk`) and the snapshot serialize loop
(`serializeSnapshot`'s `for‚Ä¶of INDEX.byEl`) were SYNCHRONOUS. They checked a
wall-clock budget inside a tight loop but never yielded ‚Äî so a ~2.5‚Äì4s budget
meant up to 2.5‚Äì4s of UNINTERRUPTED main-thread work, which is exactly what flips
a heavy renderer into "unresponsive". Worse, `withSnap` runs an auto-snapshot
after EVERY `fast_click`/`fast_fill`/etc. (`serializeSnapshot(true,{budgetMs:4000})`),
so each action re-ran the heavy walk and compounded the freeze. `fast_nav` itself
never indexed (good), but it failed because the renderer was already wedged by a
prior action's synchronous walk, so `executeScript`/probe couldn't run.

**Fix ‚Äî chunk + yield + ceiling (extension-side only):**
1. **Yielding walk.** New `yieldControl()` (MessageChannel-based ‚Äî not throttled in
   background/relay tabs, unlike `setTimeout`‚â•1s and paused `rAF`). `buildIndexSync`
   ‚Üí async `buildIndexAsync`: processes a slice, and after at most `SLICE_MS` (25ms)
   of contiguous work `await`s a yield, then resumes. Overall deadline preserved ‚Üí
   returns a partial index when it expires. Never blocks > ~25ms.
2. **Hard node-walk ceiling.** `stepInitWalk` tracks `INIT.walked` and bails the
   whole walk once `MAX_WALK` (25 000) nodes are VISITED, setting `INDEX.capped`.
   A 50k-node page never attempts a full serialize.
3. **`serializeSnapshot` is now async + yielding.** Its main loop yields every
   `SLICE_MS` and still bails at `budgetMs`. All call sites awaited (`fast_snapshot`,
   `fast_click`, `fast_fill`, `fast_fill_form`, `fast_hover`, `fast_drag`, `withSnap`).
   `executeScript` already awaits a promise-returning injected func, so the bridge
   is unchanged end-to-end.
4. **Time-boxed auto-snapshot.** `withSnap` now `await`s the yielding serialize with
   a tighter budget (`budgetMs:2000, indexMs:1500`) and, on overrun/heavy, returns
   the action result with `snapshotPartial:true` (+ `snapshotNote`) instead of
   hanging the whole call to the 30s broker timeout.
5. **Auto-degrade.** When `INDEX.capped` or the index exceeds `HUGE_INDEX` (8 000)
   entries, snapshots force viewport-only output (bounds rect-read + payload).
   Snapshot payload now carries `capped`/`partial` flags so the agent knows it's a
   degraded read and can `fast_scroll`+retry.

`fast_nav` was already non-blocking (no index on nav) ‚Äî preserved. Normal pages are
unaffected: small indexes finish within one slice and never yield, never hit the
ceiling, never degrade. **Extension-side only ‚Äî the user reloads the extension at
`chrome://extensions` to activate; NO relay deploy.** `node --check` passes on
`page.js` and all touched files.
