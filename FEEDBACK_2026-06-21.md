# FastLink — field feedback (2026-06-21)

From a long, heavy real-world session: Oracle Cloud console (JET / cross-origin + same-origin iframes), GCP console (Angular Material), Gmail, GitHub, a custom dashboard. Lots of forms, dropdowns, profile switching. Notes are ranked by impact.

## TL;DR
- It works, and DOM + trusted-events is genuinely powerful — but it was **slow**, and the slowness is almost entirely **round-trip count**, not per-call latency.
- Two changes would cut most of the pain: (1) return a **fresh post-action snapshot** from click/fill, and (2) a **DOM-first multi-step executor** (batch + waits + conditionals + verify, server-side).
- Reliability: the **service-worker disconnects** were the scariest/most time-wasting failure class.

## Why it felt slow (root cause = chattiness)
Everything is one-action-per-call over a Claude→broker→extension→Chrome→back hop. A single OCI/GCP form fill was *dozens* of sequential calls. Multipliers:
1. **The auto-snapshot returned with click/fill is the MATCH-TIME (pre-action) DOM.** To see what the action *did*, I had to fire a *second* `fast_snapshot` almost every time → ~2x the calls. This was the single biggest tax.
2. **Heavy SPAs** → `fast_snapshot` came back capped/partial/empty, forcing fallback chains: `fast_evaluate` → `fast_screenshot` → vision. Each fallback = more calls.
3. **Disconnects** → status checks, waits, retries. Roughly ~30% of total friction.
4. **Coordinate drift**: read coords → page scrolled → coords stale → re-read. Repeated cycles.
5. **Stale/cached screenshots** a couple times (same byte-identical PNG); had to bust with jpeg.

## What would have made it better (highest leverage first)
1. **Fresh post-action snapshot.** Make `fast_click`/`fast_fill`/`fast_select_option` optionally (or by default) return a *settled, post-action* snapshot, not the pre-action one. Probably halves round-trips. The current pre-action behavior is a recurring footgun.
2. **DOM-first multi-step executor.** `fast_batch` exists but is linear with no logic. Add **waits, conditionals, and per-step verification server-side** so "fill these 6 fields → set dropdown → click Next → wait for panel → read result" is ONE call returning the final state. (`fast_do` is close but vision-only and refuses to submit; a DOM-first version that can complete a flow would collapse 20 calls into 1.)
3. **Connection reliability.** Transparent auto-wake + retry instead of erroring "extension not connected." The service worker idling mid-flow (e.g. during an Oracle login) was the worst reliability issue.
4. **Atomic, scroll-safe, component-aware targeting.** A "click/fill the control associated with this label/value" that scrolls-into-view + acts in one server-side step — and that handles **custom components / visually-hidden inputs**. On OCI's JET radio-cards, coordinate clicks silently no-op'd because the real `<input type=radio>` was 0-size and detached from the visible card; programmatic `el.click()` on the hidden radio was what finally worked. Coordinate-from-prior-read also goes stale on scroll.
5. **Vision fallback returned flat not-founds on dense UIs.** `fast_locate` (Gemini) repeatedly returned `found:false` on grids of near-identical cards (shape lists, AD cards, OS tiles). Returning **top-N candidates** (or defaulting to set-of-marks / `fast_point_som`) beats a bare miss. Also hit a stale-screenshot issue feeding vision.
6. **Multi-install / profile UX.** `fast_profile` only surfaced mid-session (after a reload), and the "wrong browser / wrong profile" confusion cost real time. `fast_status` / `fast_list` should loudly state the active install + which profiles are connected, and `fast_profile` should be discoverable from the start.

## What worked well (keep / lean into)
- **Same-origin iframe reach via `fast_evaluate`** (`iframe.contentDocument`) — this is what made the GCP Angular form tractable after DOM snapshots failed.
- **Trusted clicks/typing** (`fast_click_xy` + `fast_type`, CDP Input) — React/Preact honored these where injected `.value`/synthetic clicks were ignored.
- **`fast_batch`** for known sequences; **`fast_status`** diagnostics; **`fast_nav` content-script health check**.
- Reading state via DOM (`fast_evaluate`/`fast_snapshot`) instead of vision was consistently faster and more reliable.

## Suggested north star
Treat **DOM + trusted-events as the fast path**, vision/screenshots as last resort. Optimize for **fewer round-trips**: post-action snapshots + a real server-side flow executor. Make the connection unbreakable (or invisibly self-healing). That combination would have turned tonight's slow, click-by-click form battles into a handful of calls.

## Cross-origin note (niche)
Cloud Shell is a cross-origin iframe (`shell.cloud.google.com`) — nothing reaches in (no DOM, screenshots blank). Driving terminals like that needs a different approach (or just don't — we used the CLI directly instead).
