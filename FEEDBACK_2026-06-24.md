## Pain points (running log)

- **P1 (blocker, 23:33):** No FastLink extension connected on EITHER transport at session start.
  - Local connector (`fastlink`): `connected:false`, 0 connections ever. Hint: extension service worker may have errored.
  - Cloud relay (`claude.ai Fastlink`): `connected:false`, 0 devices. Hint: extension not in "relay" mode / not paired.
  - Impact: full hard stop. Cannot navigate, snapshot, or act. The whole task is blocked on a manual user step that isn't surfaced until the agent probes.
  - Improvement idea: a single "is FastLink ready?" preflight + a crisp one-line recovery instruction up front, instead of discovering it mid-task across two separate connectors. Possibly auto-detect which connector the user intends.

- **P2 (friction, 23:35):** `developer.apple.com/account` immediately redirects to `idmsa.apple.com` sign-in, whose form (Email field, Continue, Create Account link) is rendered inside an iframe that is INVISIBLE to DOM tools.
  - `fast_snapshot` returned only the footer (Privacy Policy / Terms). `fast_scout` likewise saw only footer links. The actual login UI required a screenshot to read.
  - Impact: agent must fall back to vision/`fast_point`/`fast_fill_vision` for the entire auth flow — slower, more tokens, and `fast_fill_form` (the efficient multi-field path) is unusable.
  - Improvement idea: detect known auth-widget iframes (idmsa.apple.com, accounts.google.com already handled) and either pierce them or auto-route to vision tier without the agent having to discover the emptiness first.

- **P3 (agent-behavior miss, 23:36):** Agent took a screenshot and Read/parsed it ITSELF to comprehend the page, instead of using the Gemini vision tier (`fast_point`/`fast_locate`/`fast_fill_vision`). User flagged this directly.
  - Root tension: `fast_scout` (Gemini semantic read) returned only footer because the form is iframed, so the agent reflexively screenshotted-and-read. But the sanctioned path for iframe/opaque UI is the Gemini *vision* tools, not self-parsing.
  - Improvement idea: when `fast_scout`/`fast_snapshot` come back near-empty (only footer/chrome) but a screenshot would show content, the tooling should nudge toward `fast_point`/`fast_fill_vision` rather than leaving the agent to screenshot-and-read. A guardrail/hint on the empty-snapshot result would have prevented this.

- **P4 (friction, 23:43):** `fast_fill_vision` failed TWICE in a row with "scout gemini 503: high demand / UNAVAILABLE". The entire vision tier (`fast_fill_vision`, `fast_point`, `fast_scout`) depends on Gemini, so a Gemini 503 blocks ALL iframe/opaque-UI handling — single point of failure.
  - Impact: vision was down exactly when we needed to fill the iframed Apple account form.
  - Improvement idea: (a) auto-retry with backoff on 503 inside the tool instead of surfacing immediately; (b) a fallback vision provider; (c) cache/reuse the most recent `fast_point` coordinates so a transient 503 doesn't strand the agent.

- **P5 (MAJOR, 23:49):** The Apple "Create Your Apple Account" form is a CROSS-ORIGIN iframe (page=account.apple.com, widget=appleid.apple.com). Tested every fill path:
  - DOM (`fast_fill`/`fast_snapshot`): blocked — cross-origin SecurityError, fields invisible.
  - `fast_type`: blocked — focus-guard sees activeElement = `<iframe>` and refuses ("no editable element focused").
  - `fast_key` (raw CDP key events, shift+J): click focused the field (cursor visible) but the keystroke inserted NO text. Confirmed via screenshot — field stayed empty.
  - `fast_fill_vision`: the ONLY method that works on this iframe (its internal click + trusted-typing reaches it) — but it needs Gemini, which is 503 (P4).
  - Impact: cross-origin iframe forms are a HARD single dependency on the vision/Gemini tier. When Gemini is down they cannot be filled by ANY other FastLink primitive. The "cached-coord fallback" (W2) is a mirage for cross-origin targets.
  - Improvement ideas: (1) give `fast_type` a force/iframe mode that bypasses the activeElement guard and reuses `fast_fill_vision`'s trusted-typing internals (insertText to focused element) — would make cached-coord fills work cross-origin WITHOUT Gemini; (2) failing that, retry + OpenRouter vision fallback is the only safety net for these forms.
  - Minor accuracy notes for this form: Country/Region was already preset to "United States"; Birthday is THREE `<select>` dropdowns (Month/Day/Year), not a text field — a flat text fill of a date string was never going to work.
  - **PARTIAL RETRACTION (see P7):** the claim "only `fast_fill_vision` works" stands, but the reason it *appeared* to fail intermittently was stale-capture (P6/P7), not the iframe. With `freshCapture:true`, `fast_fill_vision` DOES fill this cross-origin iframe reliably (when Gemini is up).

- **P6 (MAJOR, 23:51):** `fast_screenshot` returns STALE/CACHED frames. Three consecutive calls returned byte-identical images (132070 bytes each) though focus moved between First/Last Name fields between captures — a live capture would differ.
  - Impact: visual verification is untrustworthy. I wrongly concluded a fill had failed because the cached frame still showed empty fields. `fast_screenshot` has NO `freshCapture` param to bust the cache (`fast_point`/`fast_fill_vision` do).
  - Improvement: add `freshCapture` to `fast_screenshot`, and/or invalidate the capture cache on any input/click/nav.

- **P7 (MAJOR, 23:51):** `fast_fill_vision` FALSE-POSITIVE. Called without `freshCapture`, it reported `filled:true` with values for all 4 fields — but a fresh `fast_point(freshCapture:true)` confirmed the fields were EMPTY (found:false ×3). It located fields off a STALE pre-warm capture and mis-targeted, so typing landed nowhere, yet still reported success because it never reads back / verifies typed text. Re-run with `freshCapture:true` → fill ACTUALLY landed (fresh `fast_point` found 'Jacob', conf 0.99).
  - Improvement: (a) `fast_fill_vision` should `freshCapture` by default (or whenever a prior action dirtied the page); (b) it MUST verify the value landed before reporting `filled:true`.
  - Note: this is the root cause behind P5's apparent intermittent failure (see P5 partial retraction).

- **P8 (friction, 23:52):** Active-tab drift. The automation target kept changing out from under us. Earlier a `fast_type` reported origin=accuweather.com (wrong tab focused). Then the Apple create-account tab (account.apple.com/account) self-navigated to account.apple.com/sign-in, losing the half-filled form. 4 tabs open (idmsa sign-in, apple sign-in, dashboard.ytx.app, accuweather).
  - Impact: long multi-step flows lose their place when focus/active-tab shifts or the page redirects.
  - Improvement: pin/operate on a specific tabId for a flow rather than "active tab"; warn when the active tab's URL changes mid-flow.

## What worked well

- **W1 (23:38):** Multi-target `fast_point` on the iframed "Create Your Apple Account" form (account.apple.com). DOM snapshot showed only nav + footer, but a SINGLE `fast_point` call with 6 targets (First name, Last name, Country/region, Birthday, Email, Password) located ALL of them via Gemini vision at confidence 1.0 — one round-trip.
  - This is the right pattern for iframe forms: multi-target `fast_point` → `fast_fill_vision`. Clean contrast with P2/P3, where the agent fumbled by self-reading a screenshot first.

- **W3 (23:51):** `freshCapture:true` is the reliable workaround for both P6 and P7 — fresh-capture `fast_point`/`fast_fill_vision` gave trustworthy results every time Gemini was up.

- **W2 (23:43) — RETRACTED (see P5):** Originally logged as "recovered from the Gemini 503 without vision by reusing cached `fast_point` coords + `fast_click_xy`/`fast_type`." This was never validated and is WRONG. `fast_click_xy` focuses the field but NO typing primitive delivers text into the cross-origin iframe input. Cached-coord fallback is a mirage for cross-origin targets. See P5.

## Improvement ideas

- **I1 (from W1, 23:38):** Make the multi-target `fast_point` → `fast_fill_vision` path the DEFAULT suggested route the moment a snapshot returns only chrome/footer. Surface it as a hint so the agent doesn't fall back to screenshot-and-read (P3).
